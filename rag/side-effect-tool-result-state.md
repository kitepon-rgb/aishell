# 副作用型toolは結果状態を返す

- 出典: AIShell代表ベンチv10（288試行）とその修正、`docs/evidence/2026-07-24-ace-070-representative-benchmark.md`、
  ADR 0017／0027、commit `1beb6e6`・`794b7ae`
- 取得日: 2026-07-24
- 確度: 高（実測。修正前後で同一タスク9試行が3/9→9/9）

## 問題

読み取り系toolは「出力＝提出する答え」なので、呼び出し側は**中継するだけ**でよい。
`read_context`は中身、`change_impact`は推奨チェック、`run_check`は合否を返し、それがそのまま成果になる。

書き換え系（副作用型）toolだけ性質が違う。成功は**副作用として**起き、返るのは領収書
（取引ID、status、ハッシュ）である。しかし呼び出し側が必要とする答えは「書き換えた結果どうなったか」で、
これはtoolの主たる出力ではない。呼び出し側は中継でなく**自分から補って**言う必要がある。

**中継は確実、自発的な補足はブレる。** ここが副作用型tool固有の弱点になる。

## 実測

AIShellの`apply_change_set`は当初、変更ごとに`change_id`・`result: "applied"`・`after_sha256`だけを返し、
書き込み後の内容を持たなかった。代表ベンチの`change-set-atomic-success`（2ファイルを原子的に適用し、
結果を`[[path, content]]`で報告するtask）で、候補は次の状態になった。

- 実ファイルは3試行とも正しく`A2\n`／`B2\n`へ書き換わっていた（sha256一致）。**仕事は正しい**
- しかしエージェントの報告は`[["a","applied"],["b","applied"]]`。**中身でなく状態を報告して0/3で失敗**

比較対象の旧版0.3.3には`apply_change_set`が無く、エージェントは`read_context`で中身を読んで
`[["src/a.txt","A2\n"]]`と報告し3/3で成功していた。**専用toolを持つ側が負ける**逆転が起きていた。

## 修正が二段階になった理由

**第一段階**（`1beb6e6`）: 小さいテキストに限り`after_content`で適用後の内容を返した。
→ 0/3から2/3へ改善したが**確実にはならなかった**。失敗した試行でも`after_content`は正しく返っており、
材料はあるのにエージェントが使わなかった。

**第二段階**（`794b7ae`）: 冗長な`result: "applied"`を削除した。
→ 3/3。ずっと落ちていた試行も`[["src/a.txt","A2\n"],…]`と正しく報告した。

第一段階だけでは足りなかったのは、応答に**答えに見えるものが2つ並んでいた**ため。
`result: "applied"`は結論の顔をしており、モデルはそちらを掴んだ。しかもこの文字列は取引全体の
`status: "committed"`と情報が完全に重複していた（commitされた取引に列挙された変更は定義上すべて適用済み）。
**情報量ゼロなのに答えの位置を占めるフィールド**が、正しい答えを押しのけていた。

## 原則

1. **副作用型toolは、副作用の結果状態を返す。** 「やった」ではなく「こうなった」を返す。
   `after_path`・`after_sha256`だけでは呼び出し側が結果を復元できない。
2. **verdictに見える冗長フィールドを置かない。** 上位のstatusから導出できる per-item の状態語は、
   情報を足さずに誤読だけを誘う。消す。
3. **token効率とは両立できる。** AIShellは小さいテキスト（4KiB以下・UTF-8）だけ内容を返し、
   大きいファイルとバイナリは従来どおりハッシュ＋diff artifactに委ねた。全件inlineにする必要はない。
4. **契約を壊さずに直せる場合がある。** 今回の変更でtool catalog digestは不変（`a7fb8c…`）だった。
   出力schemaが`changes`を汎用arrayとして宣言しており、項目を約束していなかったため。

## 波及

同種の疑いとして`change-set-stale-sha`（拒否時にどのパスが手つかずかを報告するtask）がある。
これも「拒否した結果どうなったか」を呼び出し側が補わされる形で、候補・旧版とも2/3で同点だった。
0.3.5では3/3へ回復したが、原因が同じ設計欠陥かは切り分けていない。副作用型toolの**失敗経路**についても
同じ検査をする価値がある。
