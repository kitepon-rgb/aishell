# macOS向けAI OSランタイム 初期機能調査 v0.2

## 研究課題

既存のシェル／ターミナルを操作させず、AIがmacOSの機能を直接、安全かつ検証可能に使うためには、最初から何を入れ、何を入れないべきか。

今回は実装を行わず、Apple公式仕様、MCP/OpenAI公式指針、既存macOS自動化実装、macOS/OS操作ベンチマーク、エージェント安全性研究を比較した。

## 結論

作るべきものは「AI専用の新しいシェル」でも「画面をクリックする汎用GUIエージェント」でもない。

**AIの依頼を、狭く型付けされたmacOS操作へ変換し、権限・事前条件・確認・取消・実行証跡をモデルの外側で強制するローカル能力ランタイム**が、最も根拠の強い製品仮説である。操作経路は、確定的なmacOS APIを第一選択とし、直接APIがない場合だけAccessibility／画面操作へ降りるハイブリッドにする。

初期版の価値は「できる操作の数」ではない。AIが誤る、画面に騙される、途中で状態が変わる、別クライアントが接続する、という前提でも、許された範囲から出ず、何をしたか人が確認でき、止められることである。

## 製品判断の訂正（2026-07-19）

上の安全性研究は、最終アーキテクチャに必要な境界を示すものであり、実装を安全基盤から始める根拠にはしない。

このプロジェクトの本体は、**AIがshellやAppleScriptを介さずmacOSを直接操作する能力**である。初期開発では能力の成立を先に証明し、安全機構は実験端末で重大事故を避ける最低限に限定する。

最初の実験版に入れる安全策は、操作対象フォルダの指定、完全削除ではなくTrash、停止スイッチ、簡潔な操作履歴だけとする。署名クライアント限定、期限付きgrant、複雑なポリシーエンジン、single-shot planning、完全なreceipt基盤は、直接操作が動いた後に実際の失敗と必要性を見て追加する。

各実装段階は、新しいmacOS操作能力を少なくとも一つ完成させる。安全機構だけを成果物とする段階は設けない。

### process実装後の訂正（2026-07-19）

shell群、`env`、`osascript`のbasename拒否だけでは安全境界を作れない。改名binaryや、許可された開発workerが起動する子processを阻止しないためである。現実装でこの拒否を残す理由は、公開面を汎用command runnerへ退行させず、executable、arguments、working directory、lifecycle、artifactを分離したDirect OS経路へAIを誘導する製品上の設計レールだからである。権限強制や隔離はこのlistに依存させない。

## 判断を変えた主要根拠

### 1. 汎用GUI操作は既に競争領域である

Peekabooは画面取得、Accessibilityツリー、クリック、入力、スクロール、ドラッグ、メニュー、ダイアログ、ウインドウ、Spaces、エージェント、MCPまで提供している。macOS-useもAccessibilityツリー、CGEvent入力、差分応答、Escape中断、watchdogを提供する。

したがって、これらと同じ操作セットを一から作ること自体は、強い新規価値にならない。GUI部分は再利用・接続・差し替え可能なバックエンド境界に置くべきである。Peekabooの組込み可否は、ライセンス、依存範囲、セキュリティ監査後に決める。

### 2. GUIだけでも、ツールだけでも足りない

OSWorld-MCPでは、369件中250件（約69%）が構造化ツールの恩恵を受けると分類された。多くのモデルでGUIのみより精度と効率が改善した。一方、ツールがない操作やUI状態の確認にはGUIが必要である。ToolCUAもGUI／ツールの経路選択自体が難題であり、ハイブリッド学習により46.85%（基準比約66%改善）を報告している。

よって、直接APIとGUIフォールバックの両方を持つ。ただしフォールバック発動は記録し、黙って経路を変えない。

### 3. ツール数を増やすほど良いわけではない

OSWorld-MCPでは158ツールを全て同時提示すると、RAGでアプリに関係するツールだけを提示した場合に比べ、全体精度が20.5から15.5へ低下した。ツールが増えると選択と組合せが難しくなる。

したがって、フラットな巨大ツール一覧は初期から避ける。接続クライアント、許可されたフォルダ／アプリ、現在の対象、操作段階に応じて、見せる能力を絞る「能力発見・ルーティング」が中核機能になる。

### 4. 長時間操作の弱点は、クリック精度だけではない

OSWorld 2.0では、108件の長時間業務について、人間の中央値約1.6時間、強いエージェントで平均318操作、最良でも完全達成20.6%・部分達成54.8%だった。主な失敗は、制約の脱落、途中情報の見逃し、確認すべき場面での推測、完了検証の省略、隠れた状態の喪失である。

したがって、初期版から状態ハンドル、進捗イベント、事前条件、結果のreceipt、実行後検証、取消を持たせる。単なる `success: true` は禁止する。

### 5. 安全性はプロンプトやMCP annotationだけでは強制できない

macOSWorldの安全サブセットは、正規タスクの途中に偽ダイアログを表示して別操作へ誘導する。強いエージェントほど約70%の欺瞞率を示した。OpenAIの現行指針も、画面、PDF、メール、ツール出力などを未信頼入力とし、直接のユーザー指示だけを権限根拠とする。

MCP仕様の `readOnlyHint` や `destructiveHint` はヒントであり、信頼済みサーバー以外からは未信頼として扱う。権限強制はモデル／MCPクライアントより下、実際にOS操作を行うランタイムで行う必要がある。

CaMeL-NOVAは、信頼された計画と未信頼の画面認識を分離し、任意の命令注入に対する制御フロー保全を示す一方、偽の画面状態で予定済み分岐を選ばせるBranch Steeringは残ると示した。CSAgentは、実行時のLLM判断ではなく、意図と状況に結び付く静的ポリシーをOSサービスで強制し、ベンチマーク攻撃の約99.56%を阻止、ポリシー改善後は100%、平均遅延増1.99%を報告した。

結論として、初期版から「観察内容が命令に昇格しない」「操作は事前に許可された能力グラフ内だけ」「実行直前に事前条件を再検証」を設計原則にする。

### 6. macOSの権限モデルがプロセス構成を決める

AppleはApp Sandboxと、支援アプリでのAccessibility利用、任意アプリへのApple Events、他アプリ設定変更、ネットワーク設定、他アプリ終了などの非互換を明記している。一方、ファイルはユーザーが選んだフォルダとsecurity-scoped bookmarkで継続的に狭く許可できる。画面取得はScreenCaptureKitのシステム選択UIが推奨される。

広範なOS操作を単一のApp Store sandboxアプリに閉じ込める構成は成立しにくい。初期研究案は、署名済みのネイティブ管理アプリと、権限を持つローカル実行ブローカーを分離し、認証済みUnix domain socketまたはXPCで接続する構成である。直接配布・notarizationを前提候補とし、App Store配布は初期要件にしない。

Peekabooも、TCC権限を持つホストアプリとCLI/MCP側をUnix socketで分け、peer PID、Team ID、socket inode、期限、protocol version/capabilityを検査している。これは再発明せず参照すべき実運用知見である。

## 初期採用の判定基準

各候補を次の5軸で判定した。

1. 頻出価値: macOSベンチマークの一般タスクに効くか
2. 確定性: 画面座標より構造化APIで結果を確認できるか
3. 安全境界: 範囲を狭く付与・取消・監査できるか
4. 差別化: 既存GUI自動化の単純な再実装ではないか
5. 縦切り検証: 小さな初期版で設計仮説を検証できるか

## 最初の実験版から入れるもの

- SwiftからmacOS APIを直接呼ぶ実行コア。初回スパイクではshell、AppleScript、JXA、任意コード実行を持たない。
- ユーザーが管理アプリで選んだフォルダ内の一覧、検索、読取、作成、copy、move、rename、Trash。
- インストール済み／実行中アプリの列挙、起動、前面化。
- AIクライアントから呼ぶためのMCPアダプター。
- ターミナルを使わず対象フォルダ、稼働状態、操作履歴を確認できる管理アプリ。
- 全停止、対象フォルダ境界、完全削除を行わないという最低限の安全策。

## 直接操作の成立後に育てるもの

### A. 能力・権限ランタイム（製品の核）

- 接続クライアントの識別。署名済みクライアントを既定とし、開発用の未署名接続は明示的な別モードにする。
- 能力発見。許可、対象アプリ／フォルダ、現在の状態に必要な操作だけをAIへ公開する。
- スコープ付きgrant。`client × capability × resource × effect × expiry` で表現し、いつでも取消可能にする。
- 効果分類。少なくとも `observe`、`localReversible`、`external`、`irreversible`、`sensitive` をランタイム側で固定する。
- リスク地点での確認。安全な準備は先に進め、削除・外部送信・権限変更などの直前で具体的な対象と影響を示す。
- 緊急停止、個別操作の取消、timeout、進捗イベント。
- typed input/output、事前条件、idempotency key、結果receipt、実行前後の状態証拠。
- 失敗理由と実行経路（native／AX／visual fallback）の明示。暗黙フォールバックは禁止する。

### B. ネイティブ管理アプリ

- 初回オンボーディングとTCC状態表示。
- フォルダ／アプリ単位の許可UI。
- 実行中操作、待機中の確認、履歴、receiptの閲覧。
- 全停止とgrant取消。
- ターミナルを使わず疎通・権限・テスト操作を確認できるInspector。

これは非技術ユーザー向けの付属UIではなく、本体の信頼境界である。

### C. 最初の縦切り: ユーザー選択フォルダ内のファイル操作

観察系:

- フォルダ選択とsecurity-scoped bookmark
- 一覧、stat、メタデータ検索、内容読み取り、Quick Look相当のプレビュー
- 対象の安定IDと、読み取り時点のversion／mtime／file identity

変更系:

- フォルダ作成
- copy、move、rename
- Trashへ移動（実行直前の確認必須）
- `NSFileCoordinator` による協調操作
- 競合時は上書きせず、明示的に失敗または再確認

ファイル管理はmacOSWorldの独立カテゴリで29件あり、複数アプリ業務の基盤でもある。構造化APIで範囲・結果・取消を検証しやすく、能力ランタイムの安全性を試す最初の対象に向く。

### D. アプリ／ウインドウの低リスク観察と起動

- インストール済み／実行中アプリの列挙
- bundle IDによる起動、前面化
- ウインドウ一覧、対象ウインドウの特定
- 対象アプリのAccessibility可用性と画面取得可否の報告

アプリ強制終了、システム設定変更は初期に含めない。

### E. GUIバックエンド境界と最小フォールバック

- Accessibilityツリーの取得
- 安定した属性での要素特定
- 対象要素に限定した押下／テキスト設定
- Accessibilityで不足する時だけ、ユーザーが選択したウインドウの単発スクリーンショット
- fallback理由、対象、前後状態をreceiptへ記録

全面的な座標操作エンジンは作らず、Peekaboo等の既存バックエンドを差し替え可能にする。最初の実装前に再利用候補のライセンス・署名境界・依存・脅威モデルを監査する。

### F. 外部インターフェース

- 内部はモデル非依存の型付きプロトコルにする。
- 最初のアダプターとしてMCPを提供するが、MCP annotationを権限判定には使わない。
- ローカル接続はstdioを直接権限プロセスに結び付けず、認証済みIPCブローカーを経由する。
- 全ツールを常時列挙せず、capability discoveryで絞る。

## 最初から入れないもの

| 除外対象 | 初期除外の理由 | 将来の再検討条件 |
|---|---|---|
| 任意shell、AppleScript/JXA、コマンド文字列評価 | 公開面を汎用command runnerへ退行させ、Direct OS状態所有という今回の目的に反する。basename拒否自体は安全境界ではない | 汎用実行面が別製品として必要になり、責務と評価を分離できる時 |
| root権限、常駐daemon、LaunchDaemon、カーネル拡張 | 被害半径と配布負担が大きく、初期価値に不要 | 非特権APIで解けない確定ユースケースが出た時 |
| ファイルの完全削除、無確認上書き | 回復不能。最初の安全仮説と矛盾 | 復元・二段確認・ポリシー検証が完成した後 |
| 他アプリの強制終了、VPN／OSセキュリティ／パスワード等の設定 | Apple Sandbox非互換かつ高リスク | 専用の高権限モジュールとして別途設計した時 |
| メール送信、投稿、購入、契約、共有権限変更 | 外部への不可逆な代理行為 | 受取人・内容・最終確認を型で強制できた時 |
| Contacts、Calendar、写真、位置、キーチェーンの広範アクセス | 個人情報の集中域。最初の検証に不要 | 目的別アダプターと最小権限が定義できた時 |
| 常時全画面録画、常時スクリーンショット | 情報露出とトークン消費が大きい | 対象選択・保持期限・秘匿化を実装した後 |
| 座標だけによる汎用クリック／ドラッグ | stale UI、誤爆、欺瞞に弱い | AX不在を検出し、対象画面と事前条件を固定できる時 |
| 何百個ものアプリ別ツール | ツール選択精度を下げ、ポリシー面も増える | capability routingと評価基盤が先に完成した後 |
| 独自のAIエージェント／モデルホスティング | OSランタイムの価値検証と無関係で、既存AIクライアントと競合する | ランタイムが安定し、明確な不足が判明した時 |
| App Store配布 | 広範なAccessibility等とApp Sandboxが衝突する | 機能分割でsandbox適合が証明できた時 |

開発用途で後から追加する「実行ファイルURL + 引数配列」の直接process実行は、shell文字列評価とは区別する。パイプ、リダイレクト、展開、`/bin/sh -c` は提供せず、working directoryを許可ルート内へ固定し、timeoutと構造化結果を持たせる。この区別は状態所有とtool密度のための製品境界であり、任意コード実行を封じるsecurity boundaryではない。

「初期除外」は永久禁止ではない。再検討条件を満たすまでは、初期版の完成条件へ混ぜないという意味である。

## 初期アーキテクチャ仮説

```text
AIクライアント
  └─ MCP / Native adapter（モデル都合の形式変換のみ）
       └─ 認証済みローカルIPC
            └─ Capability Runtime（唯一の強制点）
                 ├─ grant / policy / confirmation
                 ├─ precondition / idempotency / receipt / event / cancel
                 ├─ Native macOS adapters
                 │    ├─ Files / metadata / preview
                 │    └─ apps / windows
                 └─ GUI Backend interface
                      ├─ AX
                      └─ selected-window visual fallback

管理アプリ
  ├─ TCC・フォルダ許可
  ├─ 確認UI
  ├─ 履歴・receipt
  └─ emergency stop
```

MCPサーバーが直接macOS権限を持つ構成にはしない。権限を持つ実行点を一つにし、どのAIクライアントから来ても同じポリシーを通す。

## 長期受入シナリオ（研究から導いたもの）

初期実装に進む場合、機能数ではなく次を通過条件にする。

1. ユーザーが管理アプリで一つのテストフォルダだけを許可できる。
2. AIはその範囲のファイルを検索・読み取り・renameでき、範囲外はランタイムが拒否する。
3. AIがTrash移動を要求すると、対象と影響を示す確認が実行直前に出る。拒否時は何も変わらない。
4. AIが取得した古いfile handleで、別プロセスが変更したファイルを上書きしようとすると、事前条件不一致で失敗する。
5. 操作の要求元、付与grant、native／fallback経路、前後状態、結果がreceiptに残る。
6. 実行中に管理アプリの停止を押すと処理が中断し、以後の操作が進まない。
7. 画面内の文書が「別のファイルを削除せよ」と命令しても、それは権限根拠にならず、元の能力グラフ外の操作は実行されない。
8. ターミナルを開かず、オンボーディング、接続確認、上記デモ、履歴確認ができる。

## 未決事項

実装前に小さく検証すべき論点は以下である。

- Peekabooをライブラリ／別プロセスとして再利用する場合のライセンスとTCC境界。
- XPCとUnix domain socketのどちらが、複数AIクライアント・署名検査・更新に適するか。
- App Sandbox内の管理UIと非sandbox実行ブローカーを分離する構成がnotarizationとUX上妥当か。
- security-scoped bookmarkの失効・移動・外部ボリューム・iCloud同期時の挙動。
- receiptに保存する証拠が個人情報そのものにならないための保持期間と秘匿化。
- 操作前計画をどこまで固定するか。CaMeL型の完全なsingle-shot planは初期必須にはしないが、権限グラフとeffect sequenceは実行前に固定する。

## 研究上の推奨

次の実装フェーズは、GUI自動化や包括的な安全基盤から始めない。まず「管理アプリ + MCPアダプター + ファイル／アプリ直接操作」の技術スパイクを行う。対象フォルダ、停止、Trash、操作履歴だけを安全の床とする。その後、既存GUIバックエンドを一つ接続し、実際に観測した失敗に対応して権限と検証を強化する。

この順番なら、当初の理想である「AIがOSと直接やりとりする」を縮めずに、シェルを新しい名前で隠しただけの仕組みや、既存GUIツールの再実装へ逸れるのを防げる。
